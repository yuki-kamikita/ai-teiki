# 定期更新RPGの設計

定期更新RPGのコア設計を作成・改訂するときに使う。

## 要件定義フェーズ

技術選定の前に、ゲーム内で「できること」を決める要件定義を行う。
この結果で、技術選定が適切かを再確認する。

## 最初に決める方針

要件定義の最初に、ゲームの方針を決める。
- ロールプレイ重視
- 戦闘重視
- 両方しっかり

## 要件定義で決めること

- プレイヤーができること（戦闘、準備、交流、取引、掲示板など）
- 必須画面（ログイン、準備、結果、交流、管理）
- 更新サイクルと更新結果の見せ方
- 進行・成長の軸（レベル、装備、物語）
- 外部連携の必要性（メール送信、外部APIなど）
 - 世界観と戦闘ルールの入力

## クイックスタート

1. 更新サイクル範囲と更新間の想定プレイ時間を確認する。
2. 準備フェーズの行動を定義する（編成、取引、チャット、偵察）。
3. 更新フェーズを定義する（戦闘解決、報酬、シナリオ進行、状態更新）。
4. 成長要素を定義する（ステータス、アイテム、物語の解放条件、長期目標）。
5. ループを簡潔な仕様にまとめる。

## 開発開始時の提案（初回ガイダンス）

開発開始の指示が来たら、まず以下を提示する。

1. 大まかな進め方（段階）
   - 必要機能の列挙と優先順位付け
   - 技術選定（習熟度に合わせた最小構成から開始）
   - 個別機能の実装（更新処理・戦闘・ログイン・交流）
   - ゲームデータ作成（世界観、シナリオ、スキル、敵）
   - 運用準備（更新スケジュール、監視、バックアップ）
2. 最初の推奨事項
   - 「更新バッチの最小実装」を先に決める（更新の入力/出力/ログ）。
   - 「結果表示の最小画面」を先に用意する（更新結果の確認体験）。
   - 早期に小さなプレイサイクルを回し、設計を固める。
   - セキュリティとデータ保全の最低ラインを決める。

## 出力フォーマット

以下の見出しで簡潔にまとめる:
- 更新サイクル
- 更新間の活動
- 更新時の解決内容
- 成長と報酬
- ソーシャル要素
- リスクと未決事項

初級（未経験）の場合は、専門用語を避け、必ずかみ砕いて説明する。

## 実装の進め方（バックエンド不要から）

実装順は、バックエンド不要の要素から始める。
1. 必要なデータ入力（世界観、戦闘ルール）
2. トップページ
3. デザイン方針と共通CSS
4. 世界観の説明ページ
5. 戦闘ルールの説明ページ

## ドキュメント出力

決定した仕様は内容ごとに分割して.mdに出力する。
分割方針は「設計」「画面」「更新処理」「成長/報酬」「交流/告知」「管理者操作」など。
保存先は以下を基準とする。
- 開発者とAI向け: `docs/design/` 配下
- プレイヤー向け: `public/docs/` 配下

テンプレートは以下を参照する。

- `skills/teiki-game-design/references/player-doc-template.md`
