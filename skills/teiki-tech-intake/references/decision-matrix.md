## スタック選定マトリクス（定期更新RPG）

目安として使う。絶対ルールではない。

### 初級向け

- フロント: Vanilla HTML/CSS/JS
- バック: PHP（さくらのレンタルサーバー最新バージョン想定）
- DB: MySQL（レンタルサーバー標準）
- 認証: メール+パスワード（password_hash。平文不可）
- ジョブ: サーバーのcron（週次バッチ）
- ホスティング: さくらのレンタルサーバー固定

選ぶ目安:
- プログラミング未経験を含む学習目的、小規模ユーザー
- 最小ツールで高速に作りたい

リスク:
- セキュリティ見落とし、スケール限界

### 中級向け

- フロント: Next.js（React）または SvelteKit
- バック: Node.js APIルート or 別API
- DB: Postgres
- 認証: Firebase Auth / Auth0 / NextAuth
- ジョブ: マネージドcron + ワーカー
- ホスティング: Vercel + マネージドDB など

選ぶ目安:
- 小規模チーム、適度なスケール、素早い反復

リスク:
- ベンダーロックイン、運用負荷増

### 上級向け

- フロント: React or Vue + API専用バックエンド
- バック: Node.js + ジョブキュー（BullMQ, Sidekiq など）
- DB: Postgres + Redis（キュー）
- 認証: 必要に応じてOAuth/SAML、強固な自前認証
- ジョブ: 専用ワーカー群
- ホスティング: Kubernetes or マネージドコンテナ

選ぶ目安:
- 大規模チーム、高いスケールと信頼性が必要

リスク:
- 運用コスト増、構成要素が多い

### 必須事項

- パスワードの平文保存は禁止。
- バッチ更新は決定的でログを残す。
- 更新処理は再実行可能（冪等）にする。
