# 定期更新RPGの技術ヒアリング

開発者にヒアリングを行い、適切なスタックを提案するときに使う。

## ヒアリング質問（小分けで聞く）

1. 開発者レベル（初級/中級/上級）とチーム規模。
   - 初級の定義に「プログラミング未経験」を含める。
2. 対象プラットフォーム（Webのみ/将来的なネイティブ）と想定ピーク。
3. 認証要件（パスワード、マジックリンク、SNS）とセキュリティ期待。
4. 予算・ホスティング制約・好みのプロバイダ。
5. 更新サイクルと更新処理の複雑さ（簡易/複雑戦闘）。

## 選定の狙い

- スタックの複雑さを開発者レベルに合わせる。
- バッチ更新を確実に実行できる構成にする（cron/ジョブ実行/キュー）。
- 選定範囲で十分なログイン安全性を確保する。
- レンタルサーバー固定の場合はPHP前提で最小構成を選ぶ。
 - 初級（未経験）には、専門用語を使わずに平易な説明で応答する。
 - 要件定義で決めた機能が初級プランで満たせない場合は警告する。

## 出力フォーマット

以下を提示する:
- 推奨スタック（フロント、バック、DB、認証、ホスティング）
- 開発者レベルに合う理由
- リスクと対策
- 次のステップ（最小構成の雛形、検証）
 - 初級プランで満たせない要件がある場合は警告

## 開発開始時の初期提案

開発開始の指示が来たら、まず以下の形で提案する。

1. 段階別の進め方
   - 必要機能の棚卸し（ログイン、掲示板、戦闘、マップ、更新バッチ、結果表示）
   - 技術選定（習熟度・予算・運用負荷で最小構成から選ぶ）
   - 個別機能の実装（ログイン→更新バッチ→結果表示→交流→戦闘詳細）
   - ゲームデータの整備（世界観、シナリオ、スキル、敵、報酬テーブル）
   - 運用設計（更新スケジュール、失敗時の再実行、監視）
2. 最初の推奨事項
   - 更新バッチを最優先で動かせる構成を作る。
   - 認証は習熟度に応じて最小リスクの方法を選ぶ。
   - DBスキーマは「更新結果の再現性」を前提に設計する。
